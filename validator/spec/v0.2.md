# Agent Manifest Specification v0.2

## Purpose

The Agent Manifest Specification (AgentManifest) is an open protocol for making APIs discoverable, evaluable, and compensable by AI agents. It solves the three-way deadlock in the emerging agent data economy: providers can't be found or paid, developers hardcode a handful of known APIs, and agents hallucinate the gaps.

This specification defines what a compliant agent-ready API must declare about itself.

**Current version.** Includes machine-verifiable authentication and payment readiness checks, registry badges, agent operational completeness requirements, and extended validator output format.

## Manifest Location

Every compliant API **must** serve a valid manifest at:

```
/.well-known/agent-manifest.json
```

The manifest must:
- Return HTTP 200 status
- Have `Content-Type: application/json` header
- Validate against the AgentManifest JSON Schema

## Manifest Format

### Required Fields

```json
{
  "spec_version": "agentmanifest-0.2",
  "name": "string",
  "version": "string",
  "description": "string",
  "categories": ["string"],
  "primary_category": "string",
  "endpoints": [],
  "pricing": {},
  "authentication": {},
  "reliability": {},
  "agent_notes": "string",
  "contact": "string",
  "listing_requested": boolean,
  "last_updated": "ISO 8601 timestamp"
}
```

**Note**: `spec_version` must be `"agentmanifest-0.2"`.

### Field Specifications

#### spec_version
- **Type**: string
- **Required**: yes
- **Value**: Must be `"agentmanifest-0.2"`
- **Purpose**: Enables version-specific validation and parsing

#### name
- **Type**: string
- **Required**: yes
- **Min length**: 3 characters
- **Max length**: 100 characters
- **Purpose**: Human-readable name of the API

#### version
- **Type**: string (semver format)
- **Required**: yes
- **Format**: Must follow semantic versioning (e.g., "1.0.0", "2.1.3")
- **Purpose**: Tracks API version for compatibility

#### description
- **Type**: string
- **Required**: yes
- **Min length**: 100 characters
- **Purpose**: Plain language explanation written for AI agents. Should explain what the API knows and what problems it can solve. **This is the most important field for agent discovery.**

#### categories
- **Type**: array of strings
- **Required**: yes
- **Values**: Must be from controlled vocabulary (see below)
- **Purpose**: Enable category-based discovery

#### primary_category
- **Type**: string
- **Required**: yes
- **Values**: One of: `reference`, `live`, `computational`, `transactional`, `enrichment`, `personal`, `discovery`
- **Purpose**: Primary classification for filtering and search

**Primary Category Definitions:**
- **reference**: Static knowledge bases, documentation, reference data
- **live**: Real-time data sources (weather, markets, sensors)
- **computational**: APIs that perform calculations or transformations
- **transactional**: APIs that execute actions or state changes
- **enrichment**: APIs that augment existing data with additional context
- **personal**: User-specific data requiring authentication
- **discovery**: Meta-APIs for finding other data sources

#### endpoints
- **Type**: array of objects
- **Required**: yes
- **Min items**: 1
- **Purpose**: Declares available API endpoints

**Endpoint Object Format:**
```json
{
  "path": "/example",
  "method": "GET",
  "description": "Plain language description for agents",
  "parameters": [],
  "response_description": "What the response contains and how to interpret it"
}
```

**Endpoint Fields:**
- `path` (string, required): API endpoint path
- `method` (string, required): HTTP method (GET, POST, PUT, DELETE, PATCH)
- `description` (string, required, min 20 chars): When and why agents should use this endpoint
- `parameters` (array, required): Array of parameter objects or empty array
- `response_description` (string, required, min 20 chars): Response content and interpretation guide

**Parameter Object Format:**
```json
{
  "name": "ingredient",
  "type": "string",
  "required": true,
  "description": "Name of the baking ingredient"
}
```

#### pricing
- **Type**: object
- **Required**: yes
- **Purpose**: Declares pricing model and enables compensation

**Pricing Object Format:**
```json
{
  "model": "free",
  "free_tier": {
    "queries_per_day": 1000,
    "queries_per_month": 30000
  },
  "paid_tier": null,
  "support_url": "https://example.com/support"
}
```

**Pricing Fields:**
- `model` (string, required): One of: `free`, `per-query`, `subscription`, `pay-what-you-want`, `tiered`
- `free_tier` (object | null): Free usage limits
  - `queries_per_day` (number | null): Daily query limit
  - `queries_per_month` (number | null): Monthly query limit
- `paid_tier` (object | null): Paid tier details
  - `amount_usd` (number): Cost in USD
  - `unit` (string): What the cost applies to (e.g., "per 1000 queries", "per month")
  - `description` (string): Human-readable pricing description
- `support_url` (string | null): URL for voluntary contributions or payment information

**Pricing Consistency Rules:**
- If `model` is `per-query` or `subscription` or `tiered`, then `paid_tier` must not be null
- If `model` is `free`, then `free_tier` must not be null
- `support_url` if present must be a valid HTTP/HTTPS URL

#### payment (optional)
- **Type**: object
- **Required**: no (required when `prepay_required` is true)
- **Purpose**: Declares payment flow for APIs that require payment before access

**Payment Object Format:**
```json
{
  "provider": "Stripe",
  "checkout_url": "https://api.example.com/checkout",
  "key_provisioning_url": "https://api.example.com/keys",
  "accepted_methods": ["credit_card", "paypal"],
  "prepay_required": true
}
```

**Payment Fields:**
- `provider` (string | null): Payment provider name
- `checkout_url` (string, required when payment object present): URL where agents complete payment or purchase API access
- `key_provisioning_url` (string | null): URL where agents retrieve API keys after payment
- `accepted_methods` (array | null): Payment methods accepted (e.g., credit_card, crypto, paypal)
- `prepay_required` (boolean | null): Whether payment is required before API access

**Note**: AMP does not process payments, store credentials, or handle money. The validator only verifies that declared endpoints exist and respond.

#### authentication
- **Type**: object
- **Required**: yes
- **Purpose**: Declares authentication requirements

**Authentication Object Format:**
```json
{
  "required": false,
  "type": null,
  "instructions": null
}
```

**Authentication Fields:**
- `required` (boolean, required): Whether authentication is required
- `type` (string | null): One of: `api_key`, `oauth2`, `bearer`, `none` (or null if not required)
- `instructions` (string | null): Plain language instructions for obtaining credentials

**Authentication Consistency Rules:**
- If `required` is `true`, then `type` must not be null and `instructions` must not be null
- If `required` is `false`, then `type` should be `null` or `"none"`

#### reliability
- **Type**: object
- **Required**: yes
- **Purpose**: Sets expectations about API maintenance and uptime

**Reliability Object Format:**
```json
{
  "maintained_by": "individual",
  "status_url": "https://status.example.com",
  "expected_uptime_pct": 99.5
}
```

**Reliability Fields:**
- `maintained_by` (string, required): One of: `individual`, `organization`, `community`
- `status_url` (string | null): URL for status page or health monitoring
- `expected_uptime_pct` (number | null): Expected uptime percentage (0-100)

#### agent_notes
- **Type**: string
- **Required**: yes
- **Min length**: 50 characters (150 when strict validation enabled)
- **Purpose**: Freeform guidance specifically for AI agents. Should include:
  - What to know before using the API
  - Common mistakes to avoid
  - Recommended usage patterns
  - Edge cases or limitations
  - Best practices for interpretation

**Agent Operational Completeness**: agent_notes must be ≥150 characters and clearly describe:
- How to create an account (or that no account is required)
- How to obtain credentials (or that no authentication is required)
- How pricing works

The manifest must contain procedural guidance including at least the terms: "account", ("authentication" or "api key"), and "pricing". If not, validation fails with: "Manifest lacks agent-operational completeness."

#### contact
- **Type**: string
- **Required**: yes
- **Format**: Email address or HTTP/HTTPS URL
- **Purpose**: Contact information for the API maintainer

#### listing_requested
- **Type**: boolean
- **Required**: yes
- **Value**: Must be `true` for registry submission
- **Purpose**: Explicit consent for public listing

#### last_updated
- **Type**: string (ISO 8601 format)
- **Required**: yes
- **Format**: ISO 8601 timestamp (e.g., "2025-02-15T10:30:00Z")
- **Purpose**: Track manifest freshness

## Validation Process

APIs submitted to the registry undergo automated validation.

### Core Checks (always run)

1. **Manifest Reachability**: Fetch `/.well-known/agent-manifest.json` (must return 200 with `Content-Type: application/json`)
2. **Schema Validity**: Validate JSON against AgentManifest JSON Schema
3. **Spec Version Check**: Verify declared spec version is supported
4. **Description Quality**: Check minimum lengths and content quality
5. **Endpoint Reachability**: Test declared GET endpoints (where possible without auth)
6. **Pricing Consistency**: Verify pricing model matches declared tiers
7. **Category Validity**: Confirm all categories are from controlled vocabulary
8. **Authentication Consistency**: Verify auth requirements are logically consistent

### Extended Checks

9. **Authentication Verification**: Machine-verify that the declared auth flow works:
   - **api_key**: If `payment.key_provisioning_url` is declared, dry-run GET to confirm endpoint exists (200 or 401 acceptable). Does NOT require actual key issuance.
   - **oauth2**: Confirm token endpoint exists (in instructions or endpoint list) and returns 200 or 400 with OAuth structure.
   - **bearer**: Confirm instructions specify how to obtain token; confirm at least one endpoint returns 401 when unauthenticated.

10. **Payment Flow Verification** (when `payment.prepay_required === true`):
    - Confirm `checkout_url` exists and responds with HTTP 200
    - Confirm `key_provisioning_url` exists and responds with HTTP 200 or 401
    - Confirm `pricing.model !== "free"`

    The validator does NOT: process payment, submit payment, store credentials, or handle money. It only verifies endpoints exist, flow is declared, URLs resolve, and manifest fields are internally consistent.

11. **Agent Operational Completeness**: Agent notes must be ≥150 characters and include procedural guidance mentioning "account", ("authentication" or "api key"), and "pricing". Fail with "Manifest lacks agent-operational completeness." if not satisfied.

APIs that pass all checks receive a signed verification token.

## Validator Output Format

The validator response includes:

```json
{
  "url": "string",
  "validated_at": "ISO 8601",
  "passed": boolean,
  "spec_version": "string | null",
  "checks": [...],
  "verification_token": "string | null",
  "schema_valid": boolean,
  "endpoints_reachable": boolean,
  "auth_verified": boolean,
  "payment_flow_verified": boolean,
  "operationally_complete": boolean,
  "badges": ["auth-verified", "payment-ready"]
}
```

- `schema_valid`: Schema validation passed
- `endpoints_reachable`: Declared endpoints are reachable
- `auth_verified`: Auth flow verified (when strict mode and auth required)
- `payment_flow_verified`: Payment flow verified (when strict mode and prepay_required)
- `operationally_complete`: Agent notes meet operational completeness (when strict mode)
- `badges`: Informational badges (see below)

## Registry Badges

When validation runs with strict mode enabled:

- If `auth_verified === true` → add `badges: ["auth-verified"]`
- If `payment_flow_verified === true` → add `badges: ["auth-verified", "payment-ready"]`

Badges are informational only. They do NOT imply endorsement or liability. The registry may store and display badges for discoverability.

## Listing Eligibility Rules

An API **may** be listed in the AgentManifest registry if:
- It passes all validator checks
- Its manifest accurately describes its actual behavior
- It returns consistent data to validators and normal consumers

An API **must not** be listed if:
- Is designed to deceive agents about the nature or quality of its data
- Harvests personal data without declared consent mechanisms
- Provides outputs designed to cause harm to agents or users relying on them
- Returns substantively different data to validators than to normal consumers

**These rules are behavioral, not topical.** The registry does not police subject matter. It polices honesty and agent safety.

## Constraints

The validator and registry:
- Do NOT introduce centralized billing
- Do NOT store payment information
- Do NOT require human approval for validation
- All verification is stateless
- Use simple HTTP HEAD or GET checks only
- Fail gracefully with structured errors
- Remain backward compatible with v0.1 manifests
